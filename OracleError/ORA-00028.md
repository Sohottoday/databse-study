# ORA-00028

- your session has been killed
- session has been terminated

## 1. 개요

- ORA-00028은 현재 접속 중이던 Oracle 세션이 강제로 종료되었을 때 발생하는 에러이다.
- Oracle 공식 설명 기준으로, 권한을 가진 사용자가 세션을 종료했기 때문에 더 이상 해당 세션으로 데이터베이스에 로그인된 상태가 아니다.
- 해결 조치는 다시 로그인하는 것이다.

## 2. 의미

- 현재 사용 중이던 DB 접속 세션이 더 이상 유효하지 않다는 의미이다.
- 즉, 사용자가 실행 중이던 SQL, 배치, 애플리케이션 커넥션 등이 중간에 끊겼고, 해당 세션으로는 더 이상 작업을 계속할 수 없다.

## 3. 주요 발생 원인

### 3.1 DBA 또는 관리자에 의한 세션 Kill

- DBA가 다음과 같은 명령으로 세션을 강제 종료한 경우 발생할 수 있다.

```SQL
ALTER SYSTEM KILL SESSION 'sid,serial#';
```

### 3.2 장시간 수행 쿼리 또는 배치 세션 정리

- 장시간 실행 중인 쿼리, 배치, Export/Import, ETL 작업 등이 정책에 의해 종료될 수 있다.
  > 너무 오래 실행되는 SQL
  > 락을 너무 오래 잡고 있는 세션
  > 리소스를 과도하게 사용하는 세션
  > 배치 작업 중 타임아웃
  > 운영자가 문제 세션으로 판단하여 수동 종료

### 3.3 DB Profile 설정에 의한 자동 종료

- Oracle Profile에 세션 제한이 설정되어 있으면 일정 조건에서 세션이 종료될 수 있다.
- 관련 설정 예시는 다음과 같다.

```SQL
SELECT profile, resource_name, limit
FROM dba_profiles
WHERE resource_name IN (
    'IDLE_TIME',
    'CONNECT_TIME',
    'SESSIONS_PER_USER'
);
```

- IDLE_TIME : 유휴 상태로 허용되는 시간
- CONNECT_TIME : 접속 유지 가능 시간
- SESSIONS_PER_USER : 사용자별 최대 세션 수

### 3.4 애플리케이션 커넥션 풀 문제

- Java, Spring, WAS, 배치 프로그램 등에서 커넥션 풀의 오래된 세션을 재사용하다가 발생할 수 있다.

예시 로그:

```
java.sql.SQLRecoverableException: ORA-00028: your session has been killed
```

이 경우 DB 세션은 이미 종료되었지만 애플리케이션 커넥션 풀에서는 해당 커넥션을 살아있는 것으로 판단하고 재사용하면서 에러가 발생할 수 있다.

### 3.5 DB 재기동, 장애 조치, 점검 작업

다음과 같은 운영 작업 중에도 발생할 수 있다.

- DB 재기동
- 인스턴스 장애
- RAC Failover
- 패치 작업
- 점검 작업
- 네트워크 단절
- 방화벽/보안장비 세션 타임아웃

## 4. 확인 방법

### 4.1 현재 세션 확인

에러 발생 직후에는 해당 세션이 이미 종료되었을 가능성이 높다.

```SQL
SELECT sid, serial#, username, status, machin, program, module
FROM v$session
WHERE username IS NOT NULL
ORDER BY logon_time DESC;
```

### 4.2 특정 사용자 세션 확인

```SQL
SELECT sid, serial#, username, status, machin, program, logon_time
FROM v$session
WHERE username = '사용자명';
```

### 4.3 세션 Kill 이력 확인

Oracle 기본 뷰만으로는 "누가 언제 어떤 세션을 kill 했는지"가 항상 명확히 남지는 않는다.
따라서 다음 로그를 함께 확인해야 한다.

- alert log
- listener log
- AWR/ASH
- DBA 작업 이력
- 운영 배치 스크립트
- WAS 로그
- 애플리케이션 로그
- 모니터링 로그

### 4.4 Alert Log 확인

서버에ㅔ서 alert log를 확인한다

```Bash
tail -f alert_<SID>.log
```

또는 ADRCI를 사용할 수 있다.

```Bash
adrci
show alert
```

확인 포인트

- 세션 강제 종료 기록
- ORA 에러 동반 여부
- DB 재기동 여부
- 프로세스 비정상 종료 여부
- trace file 생성 여부

## 5. 조치 방법

### 5.1 단순 사용자 세션인 경우

다시 로그인 하면 된다.
Oracle 공식 조치도 다시 로그인하여 작업을 계속 하라는 내용이다.

### 5.2 애플리케이션에서 발생한 경우

커넥션 풀 설정을 확인한다.

- validation query 설정 여부
- idle timeout
- max lifetime
- connection test on borrow
- abandoned connection 처리
- DB profile의 IDLE_TIME
- WAS datasource 설정

예시 validation query

```SQL
SELECT 1 FROM dual;
```

### 5.3 배치 작업에서 발생한 경우

다음을 확인한다.

- 배치 수행 시간
- 타임아웃 설정
- 스케줄러 중복 실행 여부
- 동일 계정 세션 수 제한
- 락 대기 여부
- 임시 테이블스페이스 사용량
- Undo 사용량

### 5.4 반복 발생하는 경우

반복 발생 시 단순 재접속으로 끝내면 안 된다.
다음 정보를 수집해야 한다.

- 발생 시간 : 정확한 발생 시각
- 계정 : DB 접속 계정
- 프로그램 : SQL Developer, WAS, 배치, ETL 등
- 서버 : 접속한 WAS/배치 서버
- SQL : 실행중이던 SQL
- 세션 정보 : SID, SERIAL#
- DB 상태 : 재기동, 점검, 장애 여부
- 로그 : alert log, WAS log, batch log

## 6. 장애 분석 시나리오

### 시나리오 1. 사용자가 SQL 실행 중 ORA-00028 발생

가능성 :

- DBA가 세션을 kill
- 장시간 쿼리로 인한 강제 종료
- DB 점검 또는 재기동
- 클라이언트/네트워크 연결 문제

조치 :

1.  발생 시각 확인
2.  사용자 계정 확인
3.  DBA 작업 여부 확인
4.  alert log 확인
5.  동일 시간대 다른 사용자도 발생했는지 확인

### 시나리오 2. WAS 로그에 ORA-00028 발생

가능성 :

- 커넥션 풀의 죽은 커넥션 재사용
- DB에서 세션 종료
- 방화벽 idle timeout
- DB profile idle timeout
- WAS datasource validation 미흡

조치 :

1.  WAS 로그에서 발생 시간 확인
2.  DB alert log 확인
3.  커넥션 풀 validation 설정 확인
4.  DB profile 설정 확인
5.  방화벽 idle timeout 확인

### 시나리오 3. 배치 작업 중 ORA-00028 발생

가능성 :

- 배치 수행 시간이 너무 김
- 운영 정책상 장시간 세션 종료
- 락 대기 중 세션 종료
- DB 리소스 사용량 초과
- 스케줄러/운영자에 의한 kill

조치 :

1. 배치 시작/종료 시간 확인
2. 실패 시점의 SQL 확인
3. 동일 시간대 DB 작업 확인
4. DBA에게 세션 kill 여부 확인
5. 필요 시 배치 분할 또는 SQL 튜닝 검토

## 7. 관련 SQL

### 7.1 현재 접속 세션 조회

```SQL
SELECT sid,
    serial#,
    username,
    status,
    machine,
    program,
    module,
    logon_time
FROM v$session
WHERE username IS NOT NULL
ORDER BY logon_time DESC;
```

### 7.2 특정 계정 세션 조회

```SQL
SELECT sid,
    serial#,
    username,
    status,
    machine,
    program,
    logon_time
FROM v$session
WHERE username = '계정명';
```

### 7.3 장시간 수행 SQL 확인

```SQL
SELECT s.sid,
    s.serial#,
    s.username,
    s.status,
    s.machine,
    s.program,
    s.sql_text
FROM v$session s
JOIN v$sql q
ON s.sql_id = q.sql_id
WHERE s.username IS NOT NULL;
```

### 7.4 Profile 제한 확인

```SQL
SELECT profile,
    resource_name,
    limit
FROM dba_profiles
WHERE resource_name IN (
    'IDLE_TIME',
    'CONNECT_TIME',
    'SESSIONS_PER_USER'
)
ORDER BY profile, resource_name;
```

### 8. 예방 방법

- 장시간 수행 SQL 튜닝
- 배치 작업 시간 분산
- 커넥션 풀 validation 설정
- DB Profile 제한값 확인
- 방화벽 idle timeout 확인
- 운영자의 세션 kill 정책 문서화
- 세션 종료 발생 시 alert log와 WAS log를 함께 보관
- 반복 발생 시 발생 시간, 계정, 프로그램, SQL_ID를 기록

### 9. 정리

ORA-00028 자체는 '세션이 죽었다'는 결과 메시지에 가깝다.
중요한 것은 왜 세션이 종료되었는지를 찾는 것이다.
단순히 사용자가 SQL Developer에서 작업하다 발생했다면 재접속으로 해결될 수 있다.
하지만 WAS, 배치, ETL, 운영 프로그램에서 반복 발생한다면 커넥션 풀, DB Profile, 방화벽 타임 아웃, DBA 작업, 장시간 SQL, 락 대기 등을 함께 확인해야 한다.
