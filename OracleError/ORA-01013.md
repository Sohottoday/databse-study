# ORA-01013

ORA-01013 : user requested cancel of current operation

## 1. 개요

ORA-01013은 현재 수행중인 작업(SQL, PL/SQL, RMAN, Export, Import 등)이 사용자의 요청 또는 외부 요인에 의해 취소(Cancel)되었을 때 발생하는 오류이다.
이 오류는 데이터베이스 자체의 장애라기보다는 실행 중인 작업이 정상적으로 완료되지 못하고 중단되었음을 알리는 메시지에 가깝다.

Oracle 공식 설명은 다음과 같다.

> 현재 수행 중인 작업이 사용자의 요청에 의해 취소되었다.

## 2. 의미

ORA-01013은 Oracle이 작업을 수행하던 중 취소 신호를 받았다는 의미이다.
취소 요청은 반드시 사용자가 직접 Cancel 버튼을 누른 경우만 발생하는 것은 아니다.
다음과 같은 다양한 원인으로도 발생할 수 있다.

- 사용자가 SQL 실행 취소
- SQL Developer에서 Stop 버튼 클릭
- JDBC Query Timeout
- WAS(Query Timeout)
- 어플리케이션 Timeout
- 네트워크 단절
- RMAN 작업 중단
- Export/Import 중단
- 운영자가 세션 종료

즉, **Oracle이 작업을 완료하지 못하고 중간에 종료되었다.**
라는 의미이다.

## 3. 주요 발생 원인

### 3.1 사용자가 직접 실행 취소

가장 흔한 경우이다.

- SQL Developer Cancel
- TOAD Cancel
- DBeaver Cancel
- Ctrl + C
- JDBC Cancel

예시

```SQL
SELECT *
FROM BIG_TABLE;
```

조회가 너무 오래 걸려 사용자가 취소하면
ORA-01013이 발생한다.

### 3.2 Query Timeout

애플리케이션에서 일정 시간이 지나면 자동으로 SQL을 종료하도록 설정되어 있을 수 있다.

예시) Spring

```Java
queryTimeout = 30
```

30초 이상 수행되면 Oracle 에서는 ORA-01013이 발생할 수 있다.

### 3.3 WAS Timeout

다음과 같은 Timeout 설정에서도 발생한다.

- Tomcat
- WebLogic
- WebSphere
- JEUS

예) HTTP Timeout -> DB는 아직 실행 중 -> WAS가 SQL 취소 -> ORA-01013

### 3.4 네트워크 단절

다음 상황에서도 발생할 수 있다.

- VPN 종료
- 방화벽 Timeout
- 세션 종료
- 클라이언트 종료

### 3.5 장시간 수행 SQL

SQL이 오래 걸리면 Timeout 또는 사용자의 Cancel 가능성이 높아진다.

- Full Table Scan
- Cartesian Join
- 대량 Sort
- Hash Join
- 대용량 보고서

### 3.6 운영자가 세션 종료

DBA가 아래 명령을 수행하면 발생할 수 있다.

```SQL
ALTER SYSTEM CANCEL SQL 'SID, SERIAL#';
```

```SQL
ALTER SYSTEM KILL SESSION
```

### 3.7 RMAN / Data Pump

다음 작업에서도 발생한다.

- RMAN Backup
- RMAN Restore
- Export / Import
- Data Pump

## 4. 확인 방법

### 4.1 현재 수행 중인 SQL 확인

```SQL
SELECT sid,
    serial#,
    username,
    status,
    sql_id,
    event
FROM v$session
WHERE username IS NOT NULL;
```

### 4.2 실행 중인 SQL 확인

```SQL
SELECT sql_id,
    sql_text
FROM v$sql
WHERE sql_id='SQL_ID';
```

### 4.3 장시간 수행 SQL 확인

```SQL
SELECT sid,
    serial#,
    elapsed_seconds,
    sql_id
FROM v$session_longops
WHERE sofar <> totalwork;
```

### 4.4 Alert Log 확인

확인 항목

- 세션 종료 여부
- ORA-00600
- ORA-07445
- Instance Restart
- 프로세스 종료 여부

Linux

```Bash
tail -f alert_<SID>.log
```

### 4.5 애플리케이션 로그 확인

- Query Timeout
- Socket Timeout
- Connection Timeout
- HTTP Timeout
- JDBC Timeout

예)

```text
java.sql.SQLTimeoutException
ORA-01013
```

## 5. 조치 방법

### 5.1 사용자가 취소한 경우

특별한 조치는 필요하지 않다.

### 5.2 Query Timeout 확인

- JDBC Timeout
- Spring Query Timeout
- MyBatis Timeout
- Hibernate Timeout

필요 시 Timeout 값을 조정한다.

### 5.3 SQL 성능 개선

SQL이 오래 수행되는 경우 아래 내용을 검토한다.

- Index 확인
- Execution Plan 확인
- Full Scan 제거
- Join 개선
- Sort 감소

### 5.4 네트워크 확인

- VPN
- 방화벽
- TCP Timeout
- Idle Timeout

### 5.5 WAS 설정 확인

- Request Timeout
- Connection Timeout
- Transaction Timeout

## 6. 장애 분석 시나리오

### 6.1 시나리오 1. SQL Developer에서 ORA-01013 발생

가능성 :

- 사용자가 직접 Cancel 버튼 클릭
- SQL 수행 시간이 길어 사용자가 중단
- SQL Developer 종료

조치 :

1.  SQL 실행 시간 확인
2.  SQL 튜닝 필요 여부 검토
3.  사용자가 직접 취소했는지 확인
4.  재실행 후 동일 현상 발생 여부 확인

### 6.2 시나리오 2. WAS 로그에 ORA-01013 발생

가능성 :

- Query Timeout 초과
- Transaction Timeout
- JDBC Timeout
- HTTP Timeout
- SQL 수행 시간이 너무 김

조치 :

1.  WAS 로그 확인
2.  Timeout 설정 확인
3.  SQL_ID 확인
4.  실행 계획 확인
5.  SQL 튜닝 검토

### 6.3 시나리오 3. 특정 시간에만 반복적으로 ORA-01013 발생

가능성 :

- 백업 작업과 충돌
- 배치 작업과 충돌
- 시스템 부하 증가
- Timeout 발생

조치 :

1.  발생 시간 확인
2.  Scheduler Job 확인
3.  DB 부하 확인
4.  SQL 실행 계획 확인
5.  작업 스케줄 조정 검토


## 8. 예방 방법

- 장시간 수행 SQL 튜닝
- 적절한 Index 구성
- Execution Plan 주기적 점검
- Query Timeout 설정 검토
- WAS Timeout 설정 검토
- 네트워크 안정성 확보
- 장시간 배치 작업은 업무시간 외 수행
- 애플리케이션 로그와 DB 로그를 함께 분석

## 9. 정리

ORA-01013은 현재 수행 중인 작업이 사용자의 요청이나 외부 시스템의 Timeout에 의해 중단되었음을 의미하는 오류이다.
대부분의 경우 데이터베이스 자체의 장애는 아니며, SQL 수행 시간이 길거나 애플리케이션의 Timeout 정책 때문에 발생하는 경우가 많다.

실무에서는 "누가 작업을 취소했는가?", "왜 취소되었는가?"를 확인하는 것이 핵심이다. 단발성으로 발생했다면 재실행으로 해결되는 경우가 많지만, 반복적으로 발생한다면 SQL 성능, 애플리케이션 Timeout, 네트워크 환경 등을 종합적으로 점검해야 한다.
