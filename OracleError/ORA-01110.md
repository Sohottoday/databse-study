# ORA-01110

ORA-01110 : data file string : 'string'

> 실제 업무 중 ORA-01110이라고 발생했을 때 블록커럽션이라고 DBA님께서 확인하였으며, 비상으로 판단하여 바로 출근하였음.
> 아래의 설명과는 다르게 다른 ORA 에러 메시지와 같이 뜨지 않고 단독으로 알람 울렸음.

## 1. 개요

ORA-01110은 데이터파일(Datafile)과 관련된 오류가 발생했음을 알려주는 정보성(Informational) 메시지 이다.

이 오류는 단독으로 발생하는 경우는 거의 없으며, 대부분 다른 Oracle오류 (예: ORA-01157, ORA-01113, ORA-01122 등)와 함께 출력되어 어떤 데이터파일에서 문제가 발생했는지 알려주는 역할을 한다.

오라클 공식 설명은 다음과 같다.

> ORA-01110: data file string: 'string'
> 오류가 발생한 데이터파일 번호와 파일명을 표시하는 메시지이다.
> 즉, ORA-0110 자체가 원인이 아니라 원인 분석을 위한 힌트 라고 이해하는 것이 맞다.

## 2. 의미

Oracle이 데이터 파일 관련 오류를 감지했으며, **"문제가 발생한 데이터 파일은 이것이다."** 라는 정보를 제공하는 메시지이다.

```text
ORA-01157 : cannot indentify/lock data file 5
ORA-01110 : data file 5: '/u01/oradata/PROD/users01.dbf'
```

실제 원인은 ORA-01157이며, ORA-01110은 5번 데이터파일(users01.dbf)에 문제가 있다는 정보를 알려준다.

## 3. 주요 발생 원인

### 3.1 ORA-01157과 함께 발생(가장 흔함)

```text
ORA-01157 : cannot indentify/lock data file 5
ORA-01110 : data file 5: '/u01/oradata/PROD/users01.dbf'
```

가능성 :

- 데이터파일 삭제
- 데이터파일 이동
- 파일 시스템 장애
- Storage 장애
- Permission 변경

### 3.2 ORA-01113과 함께 발생

```text
ORA-01113 : file 7 needs media recovery
ORA-01110 : data file 7 : '/u01/oradata/PROD/example01.dbf
```

가능성 :

- 비정상 종료
- Recovery 미완료
- 복구가 필요한 데이터파일

### 3.3 ORA-01122와 함께 발생

```text
ORA-01122 : DATABASE FILE FAILED VERIFICATION CHECK
ORA-01110 : data file 3 : '/u01/oradata/PROD/sysaux01.dbf
```

가능성 :

- Control File 불일치
- 잘못된 백업 복원
- Recovery 실패

### 3.5 데이터파일 Offline 상태

파일이 Offline인데 접근하려는 경우에도 함께 출력될 수 있다.

## 4. 확인 방법

### 4.1 Alert Log 확인

가장 먼저 확인해야 한다.

- ORA-01110 바로 위 오류
- ORA-01157
- ORA-01113
- ORA-01122
- ORA-01207
- ORA-01578
- OS Error

```Bash
tail -100 alert_<SID>.log
```

### 4.2 데이터파일 상태 확인

```SQL
SELECT file#,
    name,
    status,
    enabled
FROM v$datafile;
```

### 4.3 데이터파일 존재 여부

```SQL
SELECT name
FROM v$datafile;
```

OS에서도 실제 파일이 존재하는지 확인한다.
Linux

```Bash
ls -l /u01/oradata/RPOD/
```

### 4.4 Recovery 필요 여부 확인

```SQL
SELECT *
FROM v$recover_file;
```

Recovery가 필요한 파일이 표시된다.

### 4.5 Tablespace 상태 확인

```SQL
SELECT tablespace_name,
    status
FROM dba_tablespaces;
```

### 4.6 RMAN 및 백업 확인

다음을 호가인한다.

- 최근 Restore 작업
- Recovery 수행 여부
- Backup 이력
- Control File 복원 여부

## 5. 조치 방법

### 5.1 먼저 함께 발생한 ORA 오류 확인

```text
ORA-01157
ORA-01110
```

-> ORA-01157으르 해결해야 한다.

### 5.2 데이터파일 존재 여부 확인

OS에서 실제 파일이 존재하는지 확인한다.

```text
ls -l /u01/oradata/PROD/
```

삭제되었거나 이동되었다면 원래 위치로 복구한다.

### 5.3 Recovery 수행

Recovery가 필요한 경우

```SQL
RECOVER DATAFILE 5;
```

또는

```SQL
RECOVER DATABASE;
```

### 5.4 RMAN 복원

파일이 손상되었다면

```rman
RESTORE DATAFILE 5;

RECOVER DATAFILE5;
```

### 5.5 Storage 및 파일시스템 확인

확인 항목

- Disk 장애
- SAN/NAS 장애
- Mount 해제
- 권한 변경
- Storage 연결 상태

## 6. 장애 분석 시나리오

### 6.1 시나리오 1. ORA-01157과 함께 ORA-01110 발생

가능성 :

- 데이터파일 삭제
- 데이터파일 이동
- 파일 시스템 장애
- Storage 장애

조치 :

1.  Alert Log 확인
2.  ORA-01157 내용 확인
3.  OS에서 데이터파일 존재 여부 확인
4.  Storage 상태 확인
5.  필요 시 RMAN Restore 수행

### 6.2 시나리오 2. ORA-01113과 함께 ORA-01110 발생

가능성 :

- 비정상 종료
- Recovery 미수행
- Crash Recovery 필요

조치

1.  v$recover_file 조회
2.  Recovery 필요 파일 확인
3.  RECOVER DATAFILE 수행
4.  Database Open 여부 확인

### 6.3 ORA-01122와 함께 ORA-01110 발생

가능성 :

- 데이터파일 Header 손생
- Block 손상
- Storage 오류

조치 :

1.  Alert Log 확인
2.  DBVERIFY 또는 RMAN VALIDATE 수행
3.  RMAN Restore 여부 검토
4.  Storage 장애 여부 확인

```SQL
SELECT file#,
    name,

    enabled
```

### 7.2 Recovery 대상 확인

```SQL
SELECT *
```

### 7.3 Tablespace 상태 확인

```SQL

FROM dba_tablespaces;
```

```SQL

```

```SQL

    open_mode,
    database_role
FROM v$database;
```

## 8. 예방 방법

- RMAN 백업 정기 수행
- Storage 장애 모니터링
- 데이터파일 임의 이동 금지
- 데이터파일 삭제 방지
- Control File가 Datafile 동시 백업
- Recovery 절차 정기 점검
- Alert Log 모니터링 자동화

## 9. 정리

ORA-01110은 데이터파일 자체의 오류가 아니라, 문제가 발생한 데이터파일을 알려주는 정보성 메시지이다.
따라서, ORA-01110이 발생하면 반드시 함께 출력된 다른 ORA 오류를 먼저 확인해야 한다. 실제 원인은 대부분 다른 ORA 오류이며, ORA-01110은 해당 오류가 어떤 데이터파일에서 발생했는지를 알려주는 역할을 한다.
