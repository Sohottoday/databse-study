# ORA-01555

- ORA-01555 : snapshot too old
- ORA-01555 : snapshot too old : rollback segment number 10 with name "\_~~~~" too small

## 1. 개요

'ORA-01555'는 Oracle에서 가장 많이 발생하는 오류 중 하나로, **Consistent Read(일관성 읽기)**를 수행하기 위해 필요한 Undo 정보가 이미 덮어써져 더 이상 과거 시점의 데이터를 복원할 수 없을 때 발생한다.
일반적으로 **장시간 수행되는 조회(Query)**와 **동시에 많은 DML(INSERT, UPDATE, DELETE)**이 발생하는 환경에서 자주 나타난다.

Oracle 공식 원인은 다음과 같다.

> Consistent Read에 필요한 Undo 정보가 다른 트랜잭션에 의해 덮어써졌다. 해결 방법은 UNDO_RETENTION을 증가시키거나 충분한 Undo 공간을 확보하는 것이다.

## 2. 의미

Oracle은 SELECT를 수행할 때 조회 시작 시점의 데이터를 끝까지 동일하게 보여주기 위해 Consistent Read 기능을 사용한다.
예를들어 09:00에 "SELECT \* FROM EMP;"를 실행했을 때 조회가 진행되는 동안
09:01

```SQL
UPDATE EMP
SET SAL = SAL * 2
COMMIT;
```

가 실행되어도 09시에 시작된 SELECT는 09시 당시의 데이터를 계속 보여주어야 한다.
이를 위해 Oracle은 Undo 데이터를 사용한다.
하지만 조회가 너무 오래 걸리는 동안 Undo 정보가 다른 트랜잭션에 의해 덮어써지면
Oracle은 더 이상 원래 데이터를 복원할 수 없어 ORA-01555를 발생시킨다.

### Consistent Read의 이해

Oracle의 동작 순서는 다음과 같다.
SELECT 시작 -> 조회 시작 시점 기억 -> 다른 세션에서 UPDATE 발생 -> Undo를 이용하여 과거 데이터 복원 -> Undo가 존재하면 정상 조회 -> Undo가 덮어 쓰여졌다면 ORA-01555 발생
즉, 조회 자체의 문제가 아니라 데이터를 복원할 Undo가 사라진 것이 원인이다.

## 3. 주요 발생 원인

### 3.1 장시간 수행되는 Query

가장 흔한 원인이다. 조회 시간이 길수록 ORA-01555 발생 가능성이 높아진다.

- 대용량 보고서
- 수백만 건 조회
- Full Table Scan
- 복잡한 Join
- WITH절 다수 사용
- 서브쿼리 다수 사용

### 3.2 Undo Tablespace 부족

Undo 공간이 부족하면 Oracle은 오래된 Undo부터 재사용한다.
이때 아직 조회 중인 SQL이 필요한 Undo가 삭제되면 ORA-01555가 발생한다.

### 3.3 UNDO_RETENTION 값 부족

현재 설정 확인

```SQL
SHOW PARAMETER undo_retention;
```

또는

```SQL
SELECT name, value
FROM v$parameter
WHERE name='undo_retention';
```

값이 너무 작으면 조회가 끝나기 전에 Undo가 삭제될 수 있다.

### 3.4 많은 UPDATE / DELETE

대량 DML은 Undo를 빠르게 소비한다.

- UPDATE BIG_TABLE
- DELETE BIG_TABLE
- MERGE ...

### 3.5 COMMIT을 너무 자주 수행

```SQL
LOOP
UPDATE ...
COMMIT;
END LOOP;
```

처럼 Commit을 반복하면 Undo가 빨리 재사용되어 ORA-01555 가능성이 증가한다.

### 3.6 Export / Backup

다음 작업에서도 자주 발생한다.

- Data Pump Export
- RMAN
- ETL
- 배치 프로그램

## 4. 확인 방법

### 4.1 Undo 설정 확인

```SQL
SHOW PARAMETER undo;
```

### 4.2 Undo Tablespace 확인

```SQL
SELECT tablespace_name,
    status,
    contents
FROM dba_tablespaces
WHERE contents='UNDO';
```

### 4.3 Undo 사용량 확인

```SQL
SELECT tablespace_name,
    SUM(bytes)/1024/1024 MB
FROM dba_data_files
GROUP BY tablespace_name;
```

### 4.4 Undo 통계 확인

```SQL
SELECT *
FROM v$undostat
ORDER BY begin_time DESC;
```

### 4.5 현재 실행중인 SQL

```SQL
SELECT sid,
    serial#,
    username,
    sql_id,
    event,
    status
FROM v$session
WHERE username IS NOT NULL;
```

### 4.6 실행 계획 확인

```SQL
SELECT *
FROM table(DBMS_XPLAN.DISPLAY_CURSOR);
```

## 5. 조치 방법

### 5.1 Query 튜닝

가장 좋은 해결 방법이다. 조회 시간이 줄어들면 ORA-01555 발생 가능성도 줄어든다.

- Index 추가
- Full Scan 제거
- Join 개선
- 불필요한 조회 제거

### 5.2 UNDO_RETENTION 증가

```SQL
ALTER SYSTEM
SET undo_retention=3600;
```

(1시간)
**단, Undo Tablespace 크기가 충분해야 실제로 유지된다.**

### 5.3 Undo Tablespace 확대

```SQL
ALTER DATABASE DATAFILE
'undo01.dbf'
RESIZE 5G;
```

또는 새 Datafile 추가

```SQL
ALTER TABLESPACE UNDOTBS1
ADD DATAFILE
'undo2.dbf'
SIZE 5G;
```

### 5.4 UNDO GUARANTEE 사용

Undo를 강제로 유지하는 장점이 있지만 Undo 공간 부족 시 DML이 실패할 수도 있다.

```SQL
ALTER TABLESPACE undotbs1
RETENTION GUARANTEE;
```

### 5.5 배치 시간 조정

대량 UPDATE와 대용량 조회를 동시에 수행하지 않는다.

## 6. 장애 분석 시나리오

### 6.1 시나리오 1. 대용량 조회 중 ORA-01555 발생

가능성 :

- 조회 시간이 너무 오래 걸림
- Full Table Scan 수행
- 복잡한 JOIN 또는 서브쿼리 사용
- 실행 중 다른 세션에서 대량 UPDATE/DELETE 발생
- 필요한 Undo 정보가 덮어써짐

조치 :

1.  SQL_ID 학인
2.  실행 시간 확인
3.  실행 꼐획(Execution Plan)확인
4.  Full Scan 여부 확인
5.  SQL 튜닝 가능 여부 검토
6.  동일 시간대 대량 DML 수행 여부 확인

### 6.2 시나리오 2. Data Pump Export 수행 중 ORA-01555 발생

가능성 :

- Export 수행 시간이 매우 김
- 업무 시간 중 UPDATE가 지속적으로 발생
- Undo Tablespace 부족
- UNDO_RETENTION보다 Export 시간이 길어짐

조치 :

1.  Export 수행 시간 확인
2.  Undo Tablespace 사용량 확인
3.  UNDO_RETENTION 설정 확인
4.  업무시간 대량 UPDATE 여부 확인
5.  가능하면 야간 시간대로 Export 일정 변경

### 6.3 시나리오 3. 일반 SELECT 조회 중 갑자기 ORA-01555 발생

가능성 :

- 조회 자체는 정상이나 실행 시간이 오래 걸림
- 다른 세션에서 지속적인 UPDATE/DELETE 수행
- Undo Tablespace 공간 부족
- UNDO_RETENTION 설정 부족

조치 :

1.  SQL_ID 확인
2.  실행 시간 확인
3.  v$undostat 조회
4.  Undo Tablespace 사용률 확인
5.  UNDO_RETENTION 확인
6.  SQL 튜닝 또는 Undo 공간 증설 검토

## 7. 예방 방법

- SQL 튜닝
- 조회 시간 단축
- Undo Tablespace 충분히 확보
- UNDO_RETENTION 적절히 설정
- 대량 DML과 조회 작업 분리
- Commit 남발 금지
- 대용량 배치는 야간 수행

## 8. 정리

ORA-01555는 단순히 Undo 공간이 부족해서 발생하는 오류가 아니다.
실제로는 조회 시간이 Undo 보존 시간보다 길어진 결과인 경우가 대부분이다.
따라서 SQL 튜닝, Undo 관리, 배치 스케줄 조정 이 세 가지를 함께 검토해야 근본적인 해결이 가능하다.

## 9. 심화

1.  UNDO_RETENTION만 증가시키더라도 Undo Tablespace 크기가 충분하지 않으면 Oracle은 Undo를 재사용할 수 있다. 따라서, Undo Tablespace 크기와 UNDO_RETENTION을 함께 고려해야 한다.
2.  가장 근본적인 해결 방법은 SQL 튜닝을 통해 조회 시간을 줄이는 것이다. 실행 시간이 짧아질수록 Undo 보존 시간도 줄어들기 때문이다.
3.  반복적인 COMMIT은 Undo가 더 빨리 재사용되는 환경을 만들 수 있으며, 장시간 조회가 필요한 Undo 정보를 잃게 되어 ORA-01555가 발생할 확률이 높아딘다.
